# Spec for signup-firebase-auth

branch: claude/feature/signup-firebase-auth

## Summary

Wire the existing `SignupForm` component to Firebase Authentication using the exports from `lib/firebase.ts` (`auth`, `db`). On successful signup, generate a random codename for the user by combining one word from each of three distinct word sets in PascalCase (e.g. `SilentCrimsonFox`), then set that value as the Firebase user's `displayName` and write a document to the Firestore `users` collection containing the user's `id` and `codename` (no email stored). Only the Firebase Web SDK may be used — no Admin SDK or server-side calls.

## Possible Edge Cases

- Email already in use — Firebase returns `auth/email-already-in-use`; surface a user-friendly message.
- Weak password — Firebase returns `auth/weak-password`; surface a user-friendly message.
- Network failure during `createUserWithEmailAndPassword`, `updateProfile`, or Firestore write — show a generic error and do not leave the user in a partially-created state (ideally, surface the specific step that failed).
- Word sets must each contain enough unique words that collisions are rare, but the codename does not need to be globally unique.
- Firestore write fails after auth creation — log the error; the user is still authenticated but may lack a Firestore document (acceptable for now; can be repaired on next login).
- Form submit while a request is already in-flight — disable the submit button to prevent duplicate submissions.

## Acceptance Criteria

- Submitting the signup form calls `createUserWithEmailAndPassword` from the Firebase Web SDK using the `auth` export from `lib/firebase.ts`.
- A codename is generated by randomly picking one word from each of three hard-coded word sets (adjective, colour, animal — or similar thematic groups) and joining them in PascalCase.
- `updateProfile` is called on the newly created user to set `displayName` to the generated codename.
- A document is written to the Firestore `users` collection (using `db` from `lib/firebase.ts`) with the fields `id` (Firebase UID) and `codename`; email is never stored.
- On success the user is redirected to `/heists`.
- While the signup request is in-flight the submit button is disabled and shows a loading state.
- Firebase auth errors (`auth/email-already-in-use`, `auth/weak-password`) are caught and displayed inline on the form using the existing error styling (`text-error`).
- The existing password-match validation continues to work as before.
- No server-side code or Firebase Admin SDK is introduced.

## Open Questions

- Should the three word sets be defined in the `SignupForm` component itself, or extracted to a separate utility file (e.g. `lib/codename.ts`)? separate utility.
- What happens after a successful signup — redirect to `/heists` immediately, or show a brief confirmation first? immediately.
- If the Firestore `users` write fails silently, should the user be redirected anyway or shown an error? for now log error.

## Testing Guidelines

Create a test file(s) in the ./tests folder for the new feature, and create meaningful tests for the following cases, without going too heavy:

- Codename generator returns a string in PascalCase composed of exactly three words (one from each set).
- Codename generator never returns the same word from a single set twice in a row (basic randomness check).
- Submitting the form with valid data calls `createUserWithEmailAndPassword` with the correct email and password.
- On success, `updateProfile` is called with the generated codename as `displayName`.
- On success, a Firestore document is written with `id` and `codename` fields.
- When Firebase returns `auth/email-already-in-use`, an error message is shown on the form.
- The submit button is disabled while the request is in-flight.
